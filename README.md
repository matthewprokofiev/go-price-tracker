# go-price-tracker

Парсер мониторинга цен на Go: периодически снимает цены товаров с публичного сайта,
ловит изменение и присылает уведомление в Telegram. Всю историю цен выгружает в Excel.

## Что делает

- Держит список отслеживаемых товаров в Postgres (URL + CSS-селектор цены + название).
- По расписанию обходит товары: скачивает страницу → достаёт цену → сравнивает с последней в истории.
- Если цена изменилась — записывает новую в `price_history` и шлёт в Telegram: «Товар X: было 1000 ₽, стало 900 ₽».
- Выгружает всю историю цен в `.xlsx` одной командой.

## Стек

- **Go 1.25**
- **net/http** — HTTP-запросы (таймауты, ретраи, обработка 429)
- **PuerkitoBio/goquery** — разбор HTML по CSS-селектору
- **jackc/pgx/v5** + **pgxpool** — Postgres
- **pressly/goose/v3** — миграции из `embed.FS`, применяются на старте
- **xuri/excelize/v2** — выгрузка в `.xlsx`
- **go-telegram/bot** — уведомления
- **golang.org/x/sync/errgroup** — конкурентный обход с лимитом
- **temoto/robotstxt** — уважение robots.txt
- **log/slog** — логи (текст в `local`, JSON в проде)

## Как запустить

### Через Docker Compose (рекомендуется)

```bash
cp .env.example .env      # впишите BOT_TOKEN и NOTIFY_CHAT_ID
make up                   # поднимет Postgres (ждёт healthy) и парсер, применит миграции
make down                 # остановить
```

- `BOT_TOKEN` — токен бота от [@BotFather](https://t.me/BotFather).
- `NOTIFY_CHAT_ID` — ваш Telegram id (узнать у [@userinfobot](https://t.me/userinfobot)); для группы id отрицательный.

Парсер делает **первый прогон сразу при старте**, дальше — раз в `CHECK_INTERVAL` (по умолчанию 15m).

### Локально

```bash
export DATABASE_URL="postgres://tracker:tracker@localhost:5432/tracker?sslmode=disable"
export BOT_TOKEN="..." NOTIFY_CHAT_ID="..."
make run
```

> `.env` читает только Docker Compose. При `make run` переменные задаются вручную —
> в приложении нет dotenv-загрузчика.

### Выгрузка истории в Excel

```bash
make export               # пишет prices.xlsx
# или напрямую:
go run ./cmd/parser -export prices.xlsx
```

Режим экспорта не поднимает бота — токен не нужен.

### Проверка демо-уведомления без ожидания

Цены на учебном сайте статичны, поэтому «изменение» проще всего смоделировать,
занизив последнюю цену в базе руками, — на следующем прогоне парсер увидит рост и пришлёт уведомление:

```sql
UPDATE price_history SET price = price - 100
 WHERE id = (SELECT id FROM price_history ORDER BY checked_at DESC LIMIT 1);
```

## Технические решения

Подробный разбор — в [docs/DECISIONS.md](docs/DECISIONS.md). Коротко о главном:

**Почему net/http + goquery, а не headless-браузер.** Цель — цена из HTML, а не рендер
JS-страницы. `net/http` даёт полный контроль над таймаутами, заголовками и ретраями;
goquery достаёт цену по CSS-селектору. Headless-браузер здесь — лишняя тяжёлая
зависимость. CSS-селекторы вынесены в БД рядом с товаром, чтобы менять их под чужую
вёрстку без пересборки.

**Деньги — целые копейки (`int64`), не `float64`.** Цену сравнивают на равенство
(«изменилась ли?») и хранят годами; на float это ненадёжно. `int64` даёт точное
сравнение и ровно ложится на `numeric(12,2)`.

**Ретраи и 429.** Сетевые ошибки и 5xx повторяются с экспоненциальным backoff и
jitter. Прочие 4xx (404, 403) не повторяются — от повтора не починятся. На `429`
читается `Retry-After` (секунды или HTTP-дата); есть заголовок — ждём столько,
нет — назначаем паузу сами. После 429 придерживается **весь хост**, а не только
один воркер, — чтобы не продолжать долбить сервер, который попросил остановиться.

**Почему первый прогон — до тикера.** Если уйти в `time.Ticker` сразу, первые
`CHECK_INTERVAL` минут парсер молчит, и демо выглядит сломанным. Поэтому `RunOnce`
вызывается один раз до входа в цикл. Сам цикл слушает и тикер, и `ctx.Done()`
(от `signal.NotifyContext`) — по SIGTERM завершается без утечек.

**Конкурентность на двух уровнях.** `errgroup` с `SetLimit` ограничивает ширину
прогона, а вежливость к конкретному сайту держит отдельный rate limit **на хост**
(≤2 запросов одновременно, пауза между запросами). Если все товары с одного хоста,
эффективная нагрузка на сайт — 1–2 запроса, и это правильно. Один битый товар
(упавший селектор, 404) логируется и не роняет прогон.

## Этика парсинга

Проект уважает чужой сервер:

- **robots.txt** проверяется перед каждой страницей (с кешем), объявленный `Crawl-delay` соблюдается.
- **Rate limit на хост** и пауза между запросами — нагрузку не заметит даже скромный сайт.
- **Честный User-Agent** с контактной ссылкой: админ сайта видит, кто к нему ходит, и может связаться или заблокировать именно этого бота. Маскировки под браузер нет намеренно.
- **429 не игнорируется**: при просьбе притормозить парсер ждёт и не продолжает молотить сайт.
- **Таймауты и лимит размера ответа** — ни зависший, ни аномально большой ответ не роняет процесс.

> ⚠️ **Про источник данных.** В демо используется [books.toscrape.com](https://books.toscrape.com) —
> учебная песочница, созданная специально для тренировки парсеров. Это **учебный пример**.
> Для реального клиента парсинг конкретного сайта нужно согласовывать: проверять
> Terms of Service этого сайта и применимое законодательство. Техническая возможность
> спарсить страницу не означает разрешения это делать.

## Тесты и CI

```bash
make test    # go test -race ./...
make lint    # golangci-lint (запиненная версия)
```

CI (`.github/workflows/ci.yml`) гоняет `go vet`, `golangci-lint`, `govulncheck`,
тесты под `-race` и сборку Docker-образа. Все тесты работают без сети и без Docker
(HTTP мокается через `httptest`), целевой сайт при тестах не дёргается.
